# 바이브 코딩 강의 커리큘럼 — 스토리 드리븐형

> **대상**: 개발자 + 비개발자 혼합 | **시간**: 3~8시간 (반일~풀데이 워크숍)
> **접근 방식**: 스토리 드리븐 — "AI 하나로 다 하면 왜 망하는가?"에서 시작

---

## 전체 구조 오버뷰

```
Part 1. 오프닝 — 바이브 코딩이란? (30min)
Part 2. 바이브 코딩의 진화 — 1세대 → 2세대 → 3세대 (60min)
Part 3. 하네스 엔지니어링 Deep Dive (40min)
Part 4. oh-my-agent 소개 및 세팅 (40min)
Part 5. 실습 — 슬래시 커맨드 워크플로우 (60~90min)
Part 6. 클로징 — Q&A 및 정리 (20min)
─────────────────────────────────
총 합계: 약 4~5시간 (풀데이 시 Part 5 확장)
```

## Part 1. 오프닝 — 바이브 코딩이란? (30min)

### 1-1. 아이스브레이킹 (10min)
- "여러분은 AI로 코딩해본 적 있나요?" 참여형 질문
- 바이브 코딩(Vibe Coding) 용어 소개: Andrej Karpathy의 정의
  > "코딩의 분위기에 몸을 맡기고, AI에게 만들어달라고 하면 된다"
- 라이브 데모: ChatGPT/Claude에게 "회의록 요약 앱 만들어줘" → 결과 보여주기

### 1-2. 비개발자를 위한 맥락 세팅 (10min)
- 코딩 = 사람이 기계에게 명령을 내리는 것
- 바이브 코딩 = 자연어로 AI에게 명령 → AI가 코드 생성
- 쉬운 비유: "건축가 vs 시공사 → 이제 건축주가 직접 설계하는 시대"

### 1-3. 왜 지금인가? (10min)
- 2024~2025 AI 코딩 도구 폭발적 성장
- GitHub Copilot → Cursor → Claude Code → Gemini CLI
- "코딩의 민주화" vs "잘 쓰려면 엔지니어링이 필요하다"

---

## Part 2. 바이브 코딩의 진화 — 3세대 이야기 (60min)

### 2-1. 1세대: 프롬프트 엔지니어링 (20min)

**스토리**: "마법의 주문을 외우던 시절"

- **정의**: AI에게 최적의 결과를 얻기 위해 프롬프트를 정교하게 작성하는 기술
- **핵심 기법**:
  - Few-shot prompting (예시 제공)
  - Chain-of-Thought (단계별 사고 유도)
  - Role prompting ("너는 시니어 개발자야")
- **한계점** (스토리 포인트):
  - 프롬프트가 길어질수록 AI가 "까먹음"
  - 컨텍스트 윈도우 한계 → 대규모 프로젝트 불가
  - 매번 같은 맥락을 반복 설명해야 함
- **데모**: 같은 요청을 나쁜 프롬프트 vs 좋은 프롬프트로 비교

### 2-2. 2세대: 컨텍스트 엔지니어링 (20min)

**스토리**: "AI에게 기억을 심어주자"

- **정의**: AI가 프로젝트의 맥락(코드베이스, 규칙, 히스토리)을 이해하도록 환경을 설계하는 기술
- **핵심 개념**:
  - System Prompt / Instruction Files (`CLAUDE.md`, `GEMINI.md`, `AGENTS.md` 등)
  - Rules 파일 (`.cursor/rules/`, `.github/copilot-instructions.md`)
  - MCP (Model Context Protocol) — AI에게 외부 도구 연결
  - RAG (Retrieval Augmented Generation)
- **프롬프트 엔지니어링과의 차이**:
  - 프롬프트 = 일회성 요청 최적화
  - 컨텍스트 = 지속적인 환경 최적화
- **한계점** (스토리 포인트):
  - 한 AI가 모든 영역을 다 잘할 수 없음
  - 컨텍스트가 커질수록 토큰 비용 폭증
  - "만능 AI" 접근의 구조적 한계
- **데모**: `.cursorrules` 또는 `CLAUDE.md` 설정 전 vs 후 비교

### 2-3. 3세대: 하네스 엔지니어링 (20min)

**스토리**: "AI에게 팀을 만들어주자"

- **정의**: 여러 전문 AI 에이전트를 하나의 체계(harness)로 묶어 협업시키는 기술
- **핵심 개념**:
  - Single Agent → Multi Agent
  - 역할 분리 (PM, Frontend, Backend, QA, DB...)
  - 워크플로우 오케스트레이션
  - 스킬 시스템 (재사용 가능한 전문 지식)
- **비유** (비개발자용):
  - 1세대 = 혼자 일하는 프리랜서에게 잘 설명하기
  - 2세대 = 프리랜서에게 회사 매뉴얼 주기
  - 3세대 = 전문 팀을 꾸려서 각자 역할 배분
- **데모**: oh-my-agent 아키텍처 다이어그램 보여주기

## Part 3. 하네스 엔지니어링 Deep Dive (40min)

### 3-1. 하네스 엔지니어링의 의의 (15min)
- **패러다임 전환**: "AI를 잘 쓰는 법"에서 "AI 팀을 잘 만드는 법"으로
- **개발자에게**: 코드 품질 향상, 리뷰 자동화, 일관된 아키텍처
- **비개발자에게**: AI 활용의 체계화, 재현 가능한 결과
- **조직에게**: AI 활용 표준화, 온보딩 시간 단축

### 3-2. 하네스 엔지니어링의 이점 (15min)
| 이점 | 설명 |
|------|------|
| 🎯 **역할 전문화** | 각 에이전트가 자기 도메인에 집중 → 결과 품질 ↑ |
| 💰 **토큰 효율** | 필요한 컨텍스트만 로드 → 비용 ~75% 절약 |
| 🔄 **재현성** | 워크플로우 표준화 → 누가 해도 같은 프로세스 |
| 📦 **이식성** | `.agents/` 폴더가 프로젝트와 함께 이동 |
| 🛡️ **품질 게이트** | 자동 리뷰, 보안 감사 내장 |
| 🤝 **멀티 벤더** | Claude, Gemini, Codex 등 혼용 가능 |

### 3-3. 하네스 엔지니어링 세팅 — 일반론 (10min)
- Step 1: AI IDE 선택 (Cursor, Claude Code, Antigravity 등)
- Step 2: Instruction 파일 작성 (`CLAUDE.md`, `GEMINI.md`)
- Step 3: MCP 서버 연결 (Supabase, Serena, Context7 등)
- Step 4: 워크플로우 정의 (단일 에이전트 → 멀티 에이전트)
- **문제 제기**: "이걸 매번 프로젝트마다 수동으로 하면…?"
  → oh-my-agent의 등장 배경으로 자연스럽게 연결

## Part 4. oh-my-agent 소개 및 세팅 (40min)

### 4-1. oh-my-agent란? (10min)
- Portable Multi-Agent Harness
- `.agents/` 디렉토리 하나로 모든 AI IDE에서 동작
- 14개 전문 에이전트 + 9개 워크플로우 + CLI 지원
- **비유**: oh-my-zsh가 터미널을 바꿨듯, oh-my-agent가 AI 코딩을 바꾼다

### 4-2. 설치 및 세팅 라이브 데모 (15min)
```bash
# 1줄 설치
curl -fsSL https://raw.githubusercontent.com/first-fluke/oh-my-agent/main/cli/install.sh | bash

# 또는
bunx oh-my-agent
```
- 프리셋 선택 화면 보여주기
- `.agents/` 디렉토리 구조 설명
- 핵심 구성 요소:
  - `skills/` — 에이전트별 전문 지식
  - `workflows/` — 슬래시 커맨드 정의
  - `config/` — 사용자 설정
  - `mcp.json` — MCP 연결 설정

### 4-3. 에이전트 팀 구성 이해 (15min)
- 전체 에이전트 맵 보여주기
- Orchestration 레이어 (PM, Orchestrator)
- Domain 레이어 (Frontend, Backend, DB, Mobile, Design)
- Quality 레이어 (QA, Debug)
- Utility (Translator, Commit)

## Part 5. 실습 — 슬래시 커맨드 워크플로우 (60~90min)

### 실습 시나리오: "AI 회의록 요약기 만들기"

> 참가자들이 직접 따라하며, `/brainstorm` → `/plan` → `/orchestrate` → `/review` → `/commit` 전체 워크플로우를 체험합니다.

**추천 워크플로우 순서 (상황별)**:
| 상황 | 워크플로우 순서 | 특징 |
|------|---------------|------|
| 간단한 작업 | `/plan` → `/coordinate` → `/commit` | 가이드형, 빠른 실행 |
| 품질 중시 | `/brainstorm` → `/plan` → `/ultrawork` → `/commit` | 17단계 중 11개 리뷰 |
| 대규모 병렬 | `/plan` → `/orchestrate` → `/review` → `/commit` | CLI 자동 스폰 |

> 이 실습에서는 "대규모 병렬" 패턴에 `/brainstorm`을 추가한 풀코스를 체험합니다.

### 5-1. `/brainstorm` — 아이디어 발산 (15min)

**목표**: 브레인스토밍 에이전트와 대화하며 아이디어를 구체화

```
사용자: /brainstorm
"회의록을 입력하면 AI가 자동 요약하고 액션 아이템을 추출해주는 앱을 만들고 싶어.
React + SQLite3 + Gemini API로 구성하려 해."
```

**과정 설명**:
1. 에이전트가 프로젝트 컨텍스트를 탐색 (Step 1)
2. 명확화 질문을 하나씩 물어봄 (Step 2)
3. 2~3가지 접근 방식을 제안 (Step 3)
4. 선택된 접근 방식을 상세 설계 (Step 4)
5. 설계 문서를 `docs/plans/`에 저장 (Step 5)

**포인트**: "AI가 바로 코드를 쓰는 게 아니라, 먼저 사고하고 합의한다"

### 5-2. `/plan` — 작업 계획 수립 (15min)

**목표**: PM 에이전트가 설계 문서를 기반으로 실행 가능한 태스크로 분해

```
사용자: /plan
"brainstorm에서 나온 회의록 요약기 설계를 태스크로 분해해줘"
```

**과정 설명**:
1. 설계 문서 로드
2. 기술 스택 확정
3. 태스크 분해 (우선순위, 의존성 포함)
4. API 계약 정의
5. `plan.json` 생성

**포인트**: "계획 없이 코딩하면, AI도 사람도 길을 잃는다"

### 5-3. `/orchestrate` — 자동 실행 (20~30min)

**목표**: 오케스트레이터가 plan.json을 기반으로 에이전트들을 병렬 실행

```
사용자: /orchestrate
"plan.json을 기반으로 실행해줘"
```

**과정 설명**:
1. plan.json 파싱
2. 의존성 분석 → 실행 순서 결정
3. 병렬 실행 가능한 태스크 동시 스폰
4. 각 에이전트가 맡은 영역 구현

**포인트**: "사람은 감독하고, AI 팀이 실행한다"

### 5-4. `/review` — 품질 검증 (10min)

**목표**: QA 에이전트가 구현 결과를 4가지 축으로 자동 검증

```
사용자: /review
"orchestrate 결과물을 리뷰해줘"
```

**과정 설명**:
1. 보안 감사 (OWASP Top 10)
2. 성능 분석 (번들 사이즈, Lighthouse)
3. 접근성 검증 (WCAG 2.1 AA)
4. 코드 품질 종합 보고

**포인트**: "코드가 돌아간다 ≠ 좋은 코드다"

### 5-5. `/commit` — 깔끔한 커밋 (10min)

**목표**: 커밋 에이전트가 변경 사항을 기능별로 자동 분리하여 Conventional Commits 생성

```
사용자: /commit
```

**과정 설명**:
1. 변경 사항 분석 (git diff)
2. 기능별 커밋 자동 분리
3. Conventional Commits 형식으로 메시지 생성
4. 커밋 실행

**포인트**: "커밋은 코드의 일기장이다"

### 5-6. 결과 리뷰 (10min)
- 생성된 코드 구조 확인
- 커밋 히스토리 확인 (`/commit`으로 생성된)
- 동작 데모
