# 하네스 엔지니어링의 의의

## 패러다임의 전환

하네스 엔지니어링은 단순한 도구의 발전이 아니라, <strong>"AI를 어떻게 활용할 것인가"에 대한 근본적인 사고방식의 전환</strong>입니다.

```
과거: "AI를 잘 쓰는 개인의 기술"
현재: "AI 팀을 잘 설계하는 조직의 역량"
```

## 누구에게 의미가 있는가?

### 개발자에게

| 변화 | Before (2세대) | After (3세대) |
|------|---------------|--------------|
| 코드 품질 | AI가 생성한 코드를 일일이 검토 | QA 에이전트가 자동 리뷰 |
| 아키텍처 | 매번 구조를 설명해야 함 | 스킬에 아키텍처 패턴이 내장 |
| 일관성 | 대화마다 스타일이 달라짐 | 워크플로우가 일관성 보장 |
| 온보딩 | 새 프로젝트마다 처음부터 설정 | `.agents/` 폴더 복사하면 끝 |

### 비개발자에게

- **기획자/PM**: 워크플로우(`/plan`)로 체계적인 기획 → 구현 연결
- **디자이너**: 디자인 에이전트가 디자인 시스템을 코드와 동기화
- **관리자**: AI 활용의 <strong>표준화</strong>와 <strong>재현 가능성</strong> 확보

### 조직에게

::: info 🏢 조직 차원의 의의

1. **AI 활용 표준화**: 프로젝트마다 일관된 AI 활용 체계
2. **온보딩 시간 단축**: 새 팀원이 `.agents/`만 보면 워크플로우 파악
3. **품질 게이트 자동화**: 보안, 성능, 접근성 검증이 워크플로우에 내장
4. **비용 최적화**: 토큰 효율적인 스킬 설계로 API 비용 절감
5. **벤더 독립성**: 특정 AI 도구에 종속되지 않는 이식 가능한 체계
:::

## 왜 "엔지니어링"인가?

단순히 도구를 설치하는 것이 아니라, <strong>설계가 필요</strong>하기 때문입니다:

- 어떤 에이전트가 필요한가?
- 각 에이전트에게 어떤 스킬을 부여할 것인가?
- 워크플로우를 어떤 순서로 구성할 것인가?
- 품질 게이트를 어디에 배치할 것인가?

이 <strong>설계 역량</strong>이 바이브 코딩의 결과를 결정합니다.
