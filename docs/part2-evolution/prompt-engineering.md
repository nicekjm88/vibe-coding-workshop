# 1세대: 프롬프트 엔지니어링

> **스토리**: "마법의 주문을 외우던 시절"

## 정의

<strong>프롬프트 엔지니어링(Prompt Engineering)</strong>이란, AI에게 최적의 결과를 얻기 위해 <strong>입력(프롬프트)을 정교하게 설계하는 기술</strong>입니다.

같은 질문이라도 <strong>어떻게 물어보느냐</strong>에 따라 결과가 극적으로 달라집니다.

## 핵심 기법

### 1. Few-shot 프롬프팅

AI에게 <strong>예시를 먼저 보여주고</strong> 패턴을 따르게 하는 방법입니다.

```
❌ 나쁜 예
"버튼 컴포넌트 만들어줘"

✅ 좋은 예
"아래 형식으로 버튼 컴포넌트를 만들어줘:
  
 예시)
 <Button variant="primary" size="lg">저장</Button>
 <Button variant="outline" size="sm">취소</Button>
  
 이 패턴으로 'danger' variant를 추가해줘"
```

### 2. Chain-of-Thought (단계별 사고)

AI에게 <strong>사고 과정을 단계별로 밟도록</strong> 유도합니다.

```
"TODO 앱의 인증 로직을 설계해줘.
 1단계: 필요한 API 엔드포인트를 나열하고
 2단계: 각 엔드포인트의 요청/응답 스키마를 정의하고
 3단계: 에러 핸들링 전략을 수립해줘"
```

### 3. Role 프롬프팅

AI에게 <strong>역할을 부여</strong>합니다.

```
"너는 10년 경력의 시니어 풀스택 개발자야.
 보안에 특히 신경 쓰고, 코드 리뷰를 철저히 해."
```

## 한계점

::: warning ⚠️ 프롬프트 엔지니어링의 구조적 한계

### 1. 컨텍스트 윈도우 한계
AI의 기억력에는 한계가 있습니다(토큰 수). 대화가 길어지면 초반 내용을 <strong>까먹습니다</strong>.

### 2. 매번 반복 설명
"React를 쓰고 있고, TypeScript고, Tailwind를 쓰고…" 매번 같은 맥락을 반복해야 합니다.

### 3. 대규모 프로젝트 불가
파일이 100개가 넘는 프로젝트에서는, 프롬프트만으로 전체 맥락을 전달할 수 없습니다.

### 4. 결과의 비일관성
같은 프롬프트라도 실행할 때마다 <strong>다른 스타일의 코드</strong>가 나옵니다.
:::

## 비유로 이해하기

프롬프트 엔지니어링은 마치 <strong>외국어를 잘 못하는 프리랜서</strong>에게 일을 맡기는 것과 같습니다:

- ✅ 잘 설명하면 잘 해줌
- ❌ 설명이 부족하면 엉뚱한 결과
- ❌ 지난번에 설명한 건 기억 못 함
- ❌ 매번 처음부터 다시 설명해야 함

> *"더 잘 설명하는 것만으로는 한계가 있다. AI에게 기억을 줘야 한다."*
>
> → 이것이 <strong>2세대: 컨텍스트 엔지니어링</strong>의 출발점입니다.
